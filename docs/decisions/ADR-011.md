# ADR-011 ‚Äî Drop settings.json from Security Gate Integrity Hash

**Status:** Active  
**Date:** 2026-04-06  
**Related WPs:** FIX-115, SAF-008, SAF-075

---

## Context

`security_gate.py` contains an integrity check (`verify_file_integrity()`) that SHA256-hashes both `.vscode/settings.json` and `security_gate.py` itself, comparing each against embedded constants (`_KNOWN_GOOD_SETTINGS_HASH` and `_KNOWN_GOOD_GATE_HASH`).

VS Code automatically migrates deprecated settings keys whenever a workspace is opened in a newer version of VS Code. Examples observed:
- `chat.tools.terminal.sandbox.macFileSystem` renamed to `chat.agent.sandboxFileSystem.mac`
- `chat.tools.terminal.sandbox.enabled` removed
- Key ordering changed

This modifies `.vscode/settings.json` without any user action, invalidating the embedded SHA256 hash. The result is that `verify_file_integrity()` returns `False` and **all tool calls are blocked** with a `SECURITY ALERT` message (BUG-194). This affects every freshly-created workspace and recurs whenever VS Code updates its settings schema.

`settings.json` controls only **cosmetic and discoverability settings**: `search.exclude`, `files.exclude`. These settings provide defense-in-depth against agent zone violations, but zone enforcement is handled by `security_gate.py`'s `zone_classifier` module ‚Äî not by `settings.json`. A tampered `settings.json` cannot bypass the security gate.

---

## Decision

Remove `.vscode/settings.json` from `verify_file_integrity()`. The function now only verifies `security_gate.py` against `_KNOWN_GOOD_GATE_HASH`.

Concretely:
1. `_KNOWN_GOOD_SETTINGS_HASH` constant is removed from `security_gate.py`.
2. The `verify_file_integrity()` function no longer reads or hashes `settings.json`.
3. `update_hashes.py` no longer embeds or updates a settings hash.

---

## Consequences

### Positive
- Workspaces are immune to VS Code settings migrations breaking the integrity check.
- No more `SECURITY ALERT` blocks on fresh workspace open due to settings key renaming.
- `update_hashes.py` is simpler ‚Äî only one hash to manage.

### Negative / Mitigations
- `settings.json` is no longer hash-verified. A tampered `settings.json` (e.g., removing `**/NoAgentZone` from `search.exclude`) will not be detected by the startup integrity check.
- **Mitigation:** `settings.json` only affects agent discoverability (search/file visibility). It does not control access ‚Äî zone enforcement is in `security_gate.py`. A malicious agent that removes `search.exclude` entries can discover zone names but still cannot access them (the gate denies the read/write). The security posture is unchanged for the threat model this gate defends against.
- Tests that verified settings hash behavior are updated to verify the new behavior (settings hash not checked).

---

## Alternatives Considered

- **Re-embed hash after every VS Code update:** Operationally impractical; recurs on every VS Code release.
- **Watch settings.json and auto-update hash:** Complex and introduces a new attack surface.
- **Retain settings hash with a grace-period:** Adds complexity; doesn't solve the root cause.
